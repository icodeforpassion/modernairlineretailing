From TTY to EDIFACT: Structuring Airline Messages

February 05, 2024

If you ever wondered why a single letter in an availability display can move revenue, this one is for you. From TTY to EDIFACT: Structuring Airline Messages sits at the center of how airlines sell, and the mechanics matter. I’ll keep this practical and show the moving parts without hand‑waving.

What problem are we solving?

Behind the jargon there is a straightforward goal: present a sellable option, at a price that makes sense to the customer and to the airline, and commit it reliably. The challenge is the chain of systems in between — caches, hosts, pricers, stock, and settlement — each with different views of the truth.

Mechanics, not magic

Draw the flow. Who creates the answer first? Who confirms it? What data do we reuse, and what do we recompute? Writing these steps down turns disagreements into test cases.

A concrete example

Take a simple A–B–C journey request. A seller calls search. A cache answers fast with a candidate. Before we show a price, we confirm with the airline host or offer service. Stock is checked once, at commit, and the order becomes the source of truth for fulfillment. If those steps drift, you see phantom seats, price mismatches, or carts that cannot be fulfilled.

Where teams stumble

  • Unclear ownership of truth between cache, host, and order.
  • Hidden timeouts that turn soft failures into inconsistencies.
  • Rules scattered across channels, making fixes slow.

Design moves that help

  • Expose freshness with timestamps and provenance.
  • Keep mapping to legacy explicit for faster migrations.
  • Emit events for material changes so downstream systems react.

Short checklist

  • Write the flow with sources of truth marked.
  • List the top three failure modes you see.
  • Instrument search and commit paths with IDs traceable across services.

Deep Dive: TTY, Type A, Type B & Modern Equivalents (Kerala style explanation)

Okay, let us sit down calmly (like a relaxed evening in Trivandrum with a cup of kattan chaya) and peel this onion layer by layer. No rush, no mystique.

Quick Definitions (Plain Talk)
  • TTY (Teletype / Teletypewriter): Historically the physical or emulated channel where line‑oriented uppercase messages went over a store‑and‑forward network (SITA Type B style).
  • Type A: Structured, transactional, near real-time airline messages (e.g., interactive seat confirmation) over a higher level, strongly typed network service. Lower latency, more request/response semantics.
  • Type B: Store-and-forward, line (teleprinter) style. Messages are plain text blocks with headers and codes (e.g., PNL, ADL, SSM). More tolerant of human readability; batching common.
  • EDIFACT (IATA PADIS flavor): Standardized structured segments (TAG+qualifiers+data elements). Machine-parse friendly, used for reservations, ticketing, interline, revenue accounting interfaces.
  • XML / JSON (NDC / modern APIs): Fine-grained, service-oriented, often over HTTPS. Faster iteration, more extensibility, but must still map to legacy for settlement.

Anatomy of a Type B / TTY Message

Think of it like an old postcard you drop in a box; it finds the route. Components typically:

  1. Address line(s): Destination(s) (SITA addresses, e.g., AAAAPXH). Multiple lines if multi-cast.
  2. Origin/Date-Time stamp: Often auto-added.
  3. Message type indicator: e.g., PNL, ADL, SSM.
  4. Body: Coded lines (segments, SSR/OSI elements).
  5. Ending: Sometimes // or a blank line; network-specific.
.QXHAA4D
/DELAAA
  PNL 1234/25MAY DELCOK 25MAY
  001 SMITH/JOHNMR
  002 LEENA/PRIYAMSMRS SSR WCHR
//

Note: Formatting varies by carrier/GDS and network gateway. Uppercase prevailed because of teleprinter limitations.

About PAOREQ / ITAREQ / HWPREQ (Important Disclaimer)

These label styles (REQUEST/RESPONSE pairs like PAOREQ/PAORES, ITAREQ/ITARES, HWPREQ/HWPRES) appear in several internal airline ICDs or vendor-specific EDIFACT/PADIS profiles. They are not universal AirIMP codes. So, what I provide below is a didactic template to understand the structural pattern. Please validate naming and exact segment usage with:

  • Your host provider (e.g., Amadeus, Sabre, Navitaire, etc.) ICD.
  • IATA PADIS documentation for the specific version you implement.

4. Error / Rejection Patterns

Legacy TTY often returned cryptic lines. EDIFACT enables structured errors, and modern JSON can give clear codes. One small example.

TTY Error Example

REJECT CLASS CLOSED
ALT M H ONLY
//

EDIFACT Error Snippet

RFF+CR:7K3H82'

RBD+M:H'
UNT+5+0003'

6. Practical Implementation Tips (Kerala Pragmatism)

  • Explicit Parsing: Do not rely on brittle substring offsets in TTY; use robust regex plus dictionary of recognized line prefixes.
  • Normalise Early: Convert TTY to an internal canonical object model first; then adapt to EDIFACT / JSON. Less rework later.
  • Round-Trip Tests: Serialize canonical → EDIFACT → parse back → compare. Helps catch field loss.
  • Latency Budget: Log each leg: inbound parse, validation, pricing call, response compose. Baseline before optimisation.
  • Content Validation: Build a schema for your canonical model (JSON Schema) even if original source is TTY free text.
  • Semantic Versioning: For new fields, bump a version header (e.g., custom X-VERSION) so old clients degrade gracefully.
  • Error Transparency: Map internal error codes to a stable set; avoid leaking host cryptic codes directly to API consumers.
  • Message Archive: Retain raw originals (legal/audit) + parsed JSON (analytics) with correlation IDs.

7. Validation Checklist (Quick Pass)

  • Address Validity: For each outgoing Type B, are SITA addresses correct & active?
  • Segment Conformance: EDIFACT messages validated against current PADIS dictionary? (Automate.)
  • State Synchronisation: Hold (HWPRES) always tied to a PNR locator before time-limit countdown starts?
  • RBD Consistency: All returned RBDs exist in inventory mapping table for the travel date.
  • Currency Source: Price currency = contracting currency or fallback to neutral currency rule documented?
  • Clock Discipline: Use UTC internally; present local only at edges (avoid TL confusion like IST vs +05:30 misalignment).

8. A Small Cultural Note

In Trivandrum we like to say: “First make it correct, then make it fast.” Legacy airline messaging teaches humility—if you rush modern layers without respecting old constraints, you get silent mismatches. Slow down at the start; you save plenty later.

9. Summary

Core Idea: TTY (Type B) was a low-friction human + machine compromise. EDIFACT brought structural rigor. Modern JSON/NDC brings flexibility. Your job is to anchor everything in a canonical model, maintain traceability (requestId / correlation IDs), and keep economic + operational integrity intact across translations.
